How can I use give-me-job?

This page is a practical, reader-first walkthrough of what you can do with give-me-job and the exact command to run for each scenario. For the technical, script/Task-level sequence behind each flow — aimed at contributors who already know the system — see Flows.

1. Initial configuration

Before your first run, fill profile gaps and capture search-narrowing preferences with the gap-filling interviewer, which writes config/preferences.yamlbehind a validator guard:

/gmj-interview

2. Interactive (human-in-the-loop) single-offer run

Run the whole offer→artifacts pipeline for one offer, pausing for your approval after each gate passes — the default mode:

/gmj-pipeline-run, mode human_in_the_loop

See Flows for the full single-offer pipeline sequence.

3. Fully autonomous run

Run the same pipeline unattended — mode gates only the post-PASS human pause, never the machine gates (Gate A truth, Gate B fit), which stay non-bypassable in every mode:

/gmj-pipeline-run or /gmj-batch, mode autonomous

Details in Flows.

4. Multi-offer batch

Select several shortlisted offers and run each as its own gated pipeline under one resumable batch manifest, with bounded concurrency via max_parallel_offers:

/gmj-batch

See Flows for the batch (multi-offer) flow.

5. Avoiding board bans with Firecrawl

By default gmj-offer-scout searches and fetches job boards with WebSearch/WebFetch. Setting search_provider: firecrawl inconfig/preferences.yaml is an opt-in switch — if the key is absent or not firecrawl, the defaultWebSearch/WebFetch path is unchanged. When set, every scout search/fetch call instead routes through scripts/offers/gmj_firecrawl_search.py, reducing the chance a job board detects and blocks or rate-limits the scout as a bot.

This requires a FIRECRAWL_API_KEY environment variable — the script refuses to run and never constructs a Firecrawl client without it. Switching providers changes transport only: Firecrawl calls are still fully scoped by config/sources.yaml's board/geo/language allow-list and count against the same per-board limits as WebSearch/WebFetch.

6. CV template generation from a screenshot

Paste a CV design screenshot to generate a reusable, candidate-bound HTML/Jinja2 template:

/gmj-template

7. Scheduled/unattended recurring runs

Run the autonomous pipeline unattended on a recurring schedule (cron or launchd) with a fail-closed overlap guard:

scripts/ops/gmj_cron_run.sh, driving /gmj-batch mode=autonomous

Full cron/launchd setup instructions are in Runbook.

8. Resuming an interrupted flow

Because run-scoped state lives at .pipeline/runs/<run_id>/state.json, resuming an interrupted run or batch is a pure state lookup and replay to the next step — never a restart. /gmj-runs is the read-only inspector that surfaces (but never executes) the exact resume command for any interrupted flow: a single-offer run resumes via /gmj-pipeline-run passing the existing run_id, and a batch resumes via /gmj-batch --resume <batch_id>.

/gmj-runs only prints the resume command — running it is a separate, deliberate step you take afterward.

9. Operator monitoring

Watch run/batch state live with the read-only-by-default btop-style cockpit (opt in to the mutating action layer with --manage), or inspect a terse read-only run/batch timeline:

/gmj-dashboard and /gmj-runs

See Commands for the full command surface.

10. Cleanup wizard

Free disk space from accumulated generated content with an interactive, checkbox-based wizard. It lets you pick which of the 8 fixed categories to delete (the 7 output/* subfolders plus .pipeline/runs/), shows a per-category and combined count+size summary before anything is touched, and gates every deletion behind a single, un-skippable confirm prompt that defaults to "no" — there is no bypass flag, and declining performs zero deletions:

python3 scripts/gmj_cleanup_wizard.py

This is distinct from its read-only sibling scripts/gmj_cleanup_report.py, which only reports unused-file candidates and never deletes anything.
